## Per-repo detail

### 1. rtk-ai/rtk

- **Status:** EXISTS. 70,464 stars, 4,384 forks, Apache-2.0, created 2026-01-22, pushed 2026-07-09 (3 days before this research). Not archived. `homepage: https://www.rtk-ai.app`.
- **What it is:** Single Rust binary ("Rust Token Killer") that filters/compresses ~100+ common dev command outputs (git, cargo, npm test, pytest, docker, kubectl, aws cli, terraform/pulumi, etc.) before they reach the LLM context.
- **Install:** `brew install rtk`, or `curl -fsSL https://raw.githubusercontent.com/rtk-ai/rtk/refs/heads/master/install.sh | sh` (installs to `~/.local/bin`), or `cargo install --git ...`, or prebuilt binaries from GitHub Releases.
- **Exact hook mechanism:** `rtk init -g` writes a native-binary `PreToolUse` hook (as of v0.37.2+, no shell/jq dependency — earlier installs used a `rtk-rewrite.sh` shell hook, re-run `init -g` to migrate). README quote: *"Hook-based agents rewrite Bash commands (e.g., `git status` -> `rtk git status`) before execution."* **Scope limit, explicitly documented:** the hook only intercepts the `Bash` tool — Claude Code's built-in `Read`/`Grep`/`Glob` are not touched.
- **Release artifacts:** GitHub Releases (`v0.43.0`, published 2026-06-28) ship `checksums.txt` plus `.tar.gz`/`.zip` for macOS (x86_64 + aarch64 Darwin), Linux (musl + gnu, x86_64 + aarch64), Windows (x86_64-pc-windows-msvc), and `.deb`/`.rpm` packages. **Checksums are published.**
- **Uninstall:** `rtk init -g --uninstall` (removes hook, RTK.md, settings.json entry) + `cargo uninstall rtk` / `brew uninstall rtk` for the binary itself. Clean, single documented path.
- **Claimed savings (from README table, 30-min session):** `ls`/`tree` −80%, `cat`/`read` −70%, `grep`/`rg` −80%, `git status` −80%, `git diff` −75%, `cargo test`/`npm test` −90%, aggregate **~80%** across the sampled workload. Table is explicitly labeled "Estimates based on medium-sized TypeScript/Rust projects."
- **Red flags:** None significant. Telemetry is opt-in only (device hash + aggregate command-category stats, no source/paths/args — documented in `docs/TELEMETRY.md`), overridable with `RTK_TELEMETRY_DISABLED=1`. Name-collision warning in their own README: a different "rtk" (Rust Type Kit) exists on crates.io — must `cargo install --git` explicitly.
- **Notable:** `headroomlabs-ai/headroom`'s own README explicitly ships and depends on rtk ("Headroom ships with the excellent RTK binary for shell-output rewriting... Huge thanks to the RTK team").

### 2. DietrichGebert/ponytail

- **Status:** EXISTS. 81,054 stars, 4,374 forks, MIT, created **2026-06-12** — i.e. it hit 81k stars in about **one month**. That velocity is an outlier worth independent scrutiny (Trendshift/HN-style virality is plausible in this ecosystem, but star-count manipulation is also a known pattern for viral AI tooling — flagging, not accusing).
- **Correction vs. seed catalog:** labeled `output_style` in the build prompt table. **This is not accurate to Claude Code's actual `output_style` mechanism** (a distinct, documented feature: markdown files at `~/.claude/output-styles/` activated via the `outputStyle` settings field — see mechanics section below). Ponytail is a **Claude Code plugin** that ships a `skills/` directory (SKILL.md-based, model-invoked) plus two Node.js lifecycle hooks that inject a "YAGNI ladder" ruleset into every turn. It does not touch `~/.claude/output-styles/` at all.
- **Install (Claude Code):** two separate slash commands, must be sent as separate prompts per their own README: `/plugin marketplace add DietrichGebert/ponytail` then `/plugin install ponytail@ponytail`. Requires Node.js on PATH for the lifecycle hooks (degrades gracefully to skill-only if missing).
- **Uninstall:** README states `/plugin remove ponytail` for Claude Code. **Caveat:** the official Claude Code docs (verified independently below) document the uninstall command as `/plugin uninstall <name>@<marketplace>`, not `/plugin remove`. This may be a valid shorthand alias or may be an imprecise example in ponytail's own README — worth testing empirically before Piggy automates it. The README also warns that removing the plugin leaves behind `~/.config/ponytail/config.json` and a possible `statusLine` entry in `~/.claude/settings.json`, cleaned up by `node scripts/uninstall.js` (must be run **before** the plugin-remove step, since the script is itself a plugin file).
- **Claimed savings — revised and self-corrected in-repo:** current README leads with an **agentic** benchmark (headless Claude Code session, 12 feature tickets on a real FastAPI+React repo, Haiku 4.5, n=4): **LOC −54%, tokens −22%(agentic table)/−20% cost/−27% time, 100% safety**. The README explicitly retracts its own earlier "80-94% less code" single-shot number as a "conversational-baseline artifact" after a GitHub issue (#126) called it out — a genuinely honest correction, worth noting as a positive signal.
- **Red flags:** None malicious. Requires Node.js on PATH (soft dependency, fails open). The extreme one-month star growth is worth independent verification before treating star count as a trust signal.

### 3. JuliusBrussee/caveman

- **Status:** EXISTS. 88,350 stars, 5,076 forks, MIT, created 2026-04-04, pushed 2026-07-03. Also on a rapid growth trajectory (~29k stars/month since creation) — same caveat as ponytail.
- **Correction vs. seed catalog:** same as ponytail — labeled `output_style`, but the actual mechanism is a **Claude Code skill/plugin** plus a session-start hook that writes a flag file so the persona is active from message one on Claude Code/Codex/Gemini. Not a literal `~/.claude/output-styles/` file.
- **Install:** `curl -fsSL https://raw.githubusercontent.com/JuliusBrussee/caveman/main/install.sh | bash` (auto-detects and installs for every AI agent found on the machine, ~30s, Node ≥18), or scoped: `claude plugin marketplace add JuliusBrussee/caveman && claude plugin install caveman@caveman`. Detailed per-agent uninstall matrix is in `INSTALL.md` (not fetched in this pass — the top-level README doesn't show a bare Claude Code uninstall one-liner outside the OpenClaw-specific `--uninstall` flag).
- **Claimed savings (verbatim from README):** *"why use many token when few do trick"* — **65% fewer output tokens**, benchmarked across 10 prompts (range 22–87%), output tokens only. The README itself is unusually honest about the caveat: *"Caveman only shrinks **output** tokens. Input and reasoning tokens are untouched, and the skill itself adds ~1–1.5k input tokens per turn. So whole-session savings run smaller than the output number, and on already-terse workloads they can go net-negative."* (`docs/HONEST-NUMBERS.md`).
- **JetBrains A/B verification — CONFIRMED, seed catalog claim checks out:** [JetBrains AI blog, "Speaking to AI Agents like Cavemen Saves 65% of Tokens. We Test."](https://blog.jetbrains.com/ai/2026/07/speak-to-ai-agents-like-cavemen-tosave-tokens/) (July 2026) ran a paired A/B test on **SkillsBench** using Claude Sonnet 5 (low reasoning effort) in Docker-sandboxed Harbor 0.17. Exact figures: initial 10-task smoke run showed **−29.5%**, shrinking as sample size grew, settling at **−8.5% output-token savings** across the full **82-86 paired tasks (~240 billed trials)** — i.e. roughly an 8.5% ceiling in real agentic use, versus the marketed 65% chat-only figure. Quality impact: sign test over 18 non-tied pairs, **p = 0.82**, "far from any significant difference" — no measurable quality damage. JetBrains' own conclusion, quoted: the skill is *"safe, honest about style, oversold on savings."* Independent coverage of the same test also appeared at [The New Stack](https://thenewstack.io/caveman-mode-token-savings/), [InfoWorld](https://www.infoworld.com/article/4193775/talk-like-a-caveman-prompts-save-tokens-but-far-less-than-promised.html), and [Medium](https://medium.com/@mr.curios.dev/jetbrains-tested-caveman-mode-for-ai-agents-the-result-is-useful-but-far-not-magical-2c16641cd94e).
- **Red flags:** None malicious — README explicitly states zero telemetry, zero accounts, zero network calls after install. The seed catalog's warning label is exactly correct and should ship as-is.

### 6. nadimtuhin/claude-token-optimizer

- **Status:** EXISTS. 516 stars, 61 forks, MIT, created 2025-11-10, pushed 2026-07-11. No GitHub Releases published (`releases/latest` returns 404) — distribution is npm-only. npm registry confirms `claude-token-optimizer@2.3.20`, published 2026-05-25.
- **What it is, exactly as labeled:** a `static_config` tool. `cto init` restructures a project's documentation into a 4-file "always loaded" core (`CLAUDE.md`, `COMMON_MISTAKES.md`, `QUICK_START.md`, `ARCHITECTURE_MAP.md`, ~800 tokens) plus a `.claudeignore` and archived/lazy-loaded docs (0 tokens until referenced). Framework-aware (13 frameworks auto-detected from `package.json`/`requirements.txt`/etc.).
- **Install:** `npx claude-token-optimizer init` (no install), or `curl -fsSL .../install.sh | bash` + `cto init`, or `npm install -g claude-token-optimizer`.
- **Important gap for Piggy's merge engine:** `cto` ships optional hook *templates* (`pre-tool-token-guard.sh`, `user-prompt-inject-context.sh`, etc.) but does **not** auto-merge them into `~/.claude/settings.json`. Its own README instructs: *"Copy the `settings` output into `~/.claude/settings.json` (merge with existing content)"* — i.e., the user is expected to hand-paste JSON. This is exactly the friction Piggy's "own all writes to settings.json" merge engine is designed to remove.
- **Claimed savings:** concrete, sourced anecdote — *"My RedwoodJS project was burning 11,000 tokens before I even started coding"* → reduced to ~800 tokens at session start (~1,300 total), a **~90%** reduction on that one project; README caveats *"Your mileage will vary, but 83-87% reduction is typical across frameworks."* Token counts are explicitly labeled as Claude-2-tokenizer estimates, not exact.
- **Uninstall:** no single documented uninstall command — since it only writes markdown/`.claudeignore` files (not hooks by default), removal is manual file deletion. If hooks were installed via `cto hooks install`, there's a `cto hooks status`/`cto hooks list` but no `cto hooks uninstall` shown in the README.
- **Red flags:** None. `cto init` is explicitly non-destructive (append-only unless `--force`), has a `cto audit --json` CI mode, and a `--dry-run` flag on the destructive-sounding `compress`/`prune` commands.

### 8. ooples/token-optimizer-mcp

- **Status:** EXISTS. 434 stars, 43 forks, MIT, created 2025-10-11. **GitHub pushed_at is 2026-06-23, but GitHub Releases and the npm package are both frozen at `v5.0.1`, published 2025-11-02** — i.e. ~8 months of commit activity with no corresponding release cut on either distribution channel. This is a genuine currency/maintenance-signal concern, distinct from the seed catalog's manifest-cost concern.
- **Correction/confirmation vs. seed catalog:** confirmed `context_input`/MCP-server mechanism — an MCP server exposing **65 tools** (`optimize_text`, `smart_read`, `smart_grep`, `smart_api_fetch`, `smart_cache`, 61+ more across file ops, API/DB ops, build/test, advanced caching, monitoring, system ops) plus a PowerShell/shell hook dispatcher for a "7-phase" automatic pipeline (`PreToolUse` tool replacement → cache lookups → `PostToolUse` output optimization → session tracking → `UserPromptSubmit` prompt optimization → `PreCompact` → metrics).
- **Manifest-cost claim (seed catalog: "4-6k tokens/session"):** not directly stated as a token figure anywhere in the README, but plausible given 65 distinct tool definitions with descriptions — this is exactly the class of cost Claude Code's default **MCP tool search / deferred-loading** (see mechanics section below) is designed to mitigate, *unless* this server's tools are marked `alwaysLoad`. The README doesn't mention tool-search compatibility or `alwaysLoad` either way, so the actual per-session cost depends on whether the user's Claude Code version defers this server's tool schemas — worth empirically measuring rather than trusting the seed number, but the underlying risk (many tools, thick descriptions) is real.
- **Install — and a genuine red flag:** `npm install -g @ooples/token-optimizer-mcp`. Quoting the README directly: *"That's it! The postinstall script will automatically: ... Auto-detect and configure all installed AI tools (Claude Desktop, Cursor, Cline, etc.) ... Set up automatic token optimization on every tool call."* **This means a bare `npm install -g` triggers an npm `postinstall` script that directly edits `~/.claude/settings.json` and other tools' configs with no separate opt-in/init step** — unlike rtk (`rtk init -g`, a distinct explicit step) or caveman/token-optimizer (explicit `/token-optimizer` first-run). This is a materially different trust model and the most aggressive auto-config pattern of the 8.
- **Documented bug consistent with the above:** their own Troubleshooting section describes a real, shipped bug — a UTF-8 BOM getting written into `settings.json`, breaking Claude Code with an "Invalid Settings" error, fixed in v3.0.2+ with manual BOM-stripping `sed`/PowerShell commands provided as a workaround.
- **Uninstall — genuine gap:** searched the entire 815-line README for "uninstall" — **zero matches**. There is no documented uninstall procedure at all. A user (or Piggy's install engine) would have to manually locate and strip the hook entries from `~/.claude/settings.json`, delete `~/.claude-global/hooks/`, and remove `~/.token-optimizer/cache.db` by hand. This directly conflicts with Piggy's stated non-negotiable principle #3 ("Everything reversible. Every install has a tested uninstall.").
- **Claimed savings:** "60-90% token reduction across 38,000+ operations in real-world usage," with a headline "95%+" in the repo description that the README's own detail tables don't fully substantiate (per-category tables show 60-90%, 83-95%, 83-90%, 70-85% — the "95%+" appears to be a best-case ceiling, not the typical figure). Token counting explicitly uses the GPT-4 tokenizer as an approximation for Claude, acknowledged in their own Limitations section.
- **Verdict:** given the stale releases, the no-opt-in auto-config pattern, the documented settings.json-corruption bug, and the complete absence of an uninstall path, this is the weakest of the 8 on trust/reversibility grounds — recommend v2 at earliest, and only after Piggy's own health-check/rollback system is proven robust enough to compensate for the tool's own lack of one.

## Claude Code config mechanics (verified against code.claude.com/docs, July 2026)

### Hooks (`~/.claude/settings.json` / `.claude/settings.json`)

Top-level shape:

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          { "type": "command", "command": "path/to/script.sh", "timeout": 30 }
        ]
      }
    ]
  }
}
```

- **Event names actually exist for:** `SessionStart`, `Setup`, `SessionEnd`, `UserPromptSubmit`, `UserPromptExpansion`, `Stop`, `StopFailure`, `PreToolUse`, `PostToolUse`, `PostToolUseFailure`, `PostToolBatch`, `PermissionRequest`, `PermissionDenied`, `SubagentStart`, `SubagentStop`, `TaskCreated`, `TaskCompleted`, `TeammateIdle`, `PreCompact`, `PostCompact`, `Notification`, `MessageDisplay`, `ConfigChange`, `CwdChanged`, `FileChanged`, `WorktreeCreate`, `WorktreeRemove`, `InstructionsLoaded`, `Elicitation`, `ElicitationResult`. **There is no event literally named `context_input`** — that label is purely the seed catalog's own internal taxonomy for "reduces what enters context," not a real Claude Code API name. Not a problem, just worth knowing it's Piggy's vocabulary, not Anthropic's.
- **`matcher` field:** `"*"`/`""`/omitted = match all; alphanumeric/`_`/`-`/space/`,`/`|` = exact string or pipe-separated list (e.g. `Bash`, `Edit|Write`); anything else = unanchored regex (e.g. `mcp__memory__.*`).
- **Hook handler `type` values:** `command` (shell command/script, supports `args`, `async`, `shell`, `if` permission-rule gate, `timeout`, `statusMessage`), `http` (webhook with `url`/`headers`/`allowedEnvVars`), `mcp_tool` (invoke a configured MCP tool directly), `prompt` (ask a model to evaluate, cheap gate), `agent` (run a subagent for the check).
- **Priority order for hook definitions:** `~/.claude/settings.json` (user) → `.claude/settings.json` (project) → `.claude/settings.local.json` (project-local, gitignored) → managed policy settings (admin) → plugin `hooks/hooks.json` → skill/agent frontmatter `hooks:` section. **This directly matters for Piggy's merge engine ordering requirement** (`ordering` field in the registry sketch) — Piggy writing to user-level `settings.json` will be overridden by anything a project sets locally.

### Output styles (confirmed distinct from what ponytail/caveman actually do)

- **File location:** `~/.claude/output-styles/` (user), `.claude/output-styles/` (project, loaded from every nested dir between cwd and repo root), or inside a plugin's `output-styles/` directory.
- **Format:** Markdown file with YAML frontmatter (`name`, `description`, `keep-coding-instructions`, `force-for-plugin`) followed by the instructions appended to the system prompt.
- **Activation:** `/config` → **Output style**, which writes an `outputStyle` field to `.claude/settings.local.json`; can also be set directly: `{"outputStyle": "Explanatory"}`. **The standalone `/output-style` command was deprecated in v2.1.73 and removed in v2.1.91** — worth knowing if any Piggy documentation or health-check assumes that command still exists.
- **Built-ins:** Default, Proactive, Explanatory, Learning.
- **Why this matters for the registry:** genuinely modifies the system prompt (persistent, cached, applies to every response) — architecturally different from a skill (model-invoked, loads only when relevant) or a hook (fires on specific events). Neither ponytail nor caveman uses this mechanism, so relabeling them `claude_plugin` (not `output_style`) in Piggy's registry is not just pedantic — it changes what a health check should actually verify (presence of an installed plugin + its hooks, not an `outputStyle` settings field).

### Plugins / skills / MCP servers

- **Plugin install:** `/plugin marketplace add <owner/repo | git-url | local-path | marketplace.json-url>`, then `/plugin install <plugin-name>@<marketplace-name>`. Non-interactive equivalent: `claude plugin install <name>@<marketplace> [--scope user|project|local]`.
- **Plugin manage:** `/plugin list [--enabled|--disabled]`, `/plugin disable <name>@<marketplace>` (keeps installed, turns off), `/plugin enable <name>@<marketplace>`, `/plugin uninstall <name>@<marketplace>` (full removal). Changes need `/reload-plugins` to take effect without restarting.
- **Where plugin state lives:** installs write to `~/.claude/settings.json` (user scope) or `.claude/settings.json` (project scope, shared via version control) or `.claude/settings.local.json` (local scope, gitignored) depending on the scope chosen at install time, under an `enabledPlugins`-style field (see `/en/settings#plugin-settings`) plus `extraKnownMarketplaces` for team-configured marketplaces.
- **Plugin bundling:** a single plugin directory can ship `skills/` (SKILL.md folders), `agents/`, `hooks/hooks.json`, `.mcp.json` (MCP servers), `.lsp.json`, `monitors/monitors.json`, `bin/` (added to Bash's PATH while enabled), and its own default `settings.json` — this is exactly the shape that rtk, ponytail, caveman, alexgreensh/token-optimizer, and context-mode all use, confirming that `claude_plugin` is the dominant real `installType` for this category, not the finer-grained `skill`/`output_style`/`mcp_server` buckets the seed registry schema separates them into.
- **Skills standalone (non-plugin):** `.claude/skills/<name>/SKILL.md` (project) or `~/.claude/skills/<name>/SKILL.md` (user) — invoked as `/skill-name`, or `/plugin-name:skill-name` when shipped inside a plugin (namespaced to prevent collisions).
- **MCP servers:** three scopes — **local** (default, private to you + current project, stored in `~/.claude.json` under that project's path), **project** (`.mcp.json` at repo root, shared via VCS, requires one-time approval per teammate — `claude mcp reset-project-choices` to reset), **user** (`~/.claude.json`, all your projects). Add via `claude mcp add --transport http|sse|stdio|ws <name> <url-or-command>` or `claude mcp add-json <name> '<json>'`. Manage: `claude mcp list`, `claude mcp get <name>`, `claude mcp remove <name>`, or `/mcp` inside a session. **Precedence when duplicated:** local > project > user > plugin-provided > claude.ai connectors.
- **MCP context-cost mitigation (directly relevant to the `token-optimizer-mcp` 4-6k-token manifest-cost concern):** "tool search" is **on by default** — MCP tool schemas are deferred and only loaded into context when Claude's `ToolSearch` call finds them relevant, rather than all being loaded upfront at session start. A server can opt out of deferral per-server (`"alwaysLoad": true` in its `.mcp.json` entry) or per-tool (`"anthropic/alwaysLoad": true` in a tool's `_meta`). Whether `ooples/token-optimizer-mcp`'s 65 tools pay their claimed 4-6k-token tax on every session therefore depends on whether that project's server config sets `alwaysLoad` — not stated either way in its README, and worth an empirical context-window measurement rather than trusting either the seed catalog's number or the absence of a stated one.
- **Disabling an MCP server without removing config:** no dedicated "disable" verb for user/project/local-scoped servers beyond `claude mcp remove`; for **claude.ai connectors** specifically there's a dedicated `disableClaudeAiConnectors: true` settings flag and a `deniedMcpServers` managed-settings list for blocking by name/URL pattern.

## Summary of corrections to the seed catalog

1. **Existence: no corrections needed.** All 8 repos are real, live, and match their claimed descriptions closely. This seed catalog is unusually reliable — treat that as a data point, not an assumption to carry into future catalog refreshes.
2. **Taxonomy: `output_style` is wrong for ponytail and caveman.** Both are Claude Code **plugins** (skills + hooks), not `~/.claude/output-styles/` files. Recommend relabeling `installType: claude_plugin` for both, and treating `output_style` as reserved for tools that literally ship a markdown file to that directory (none of the 8 do).
3. **License risk, not flagged in the seed catalog:** `alexgreensh/token-optimizer` (PolyForm Noncommercial) and `mksglu/context-mode` (Elastic License 2.0) are both source-available, not OSI-permissive. Worth a deliberate call before bundling/recommending them from a stated "free, open-source" app.
4. **`ooples/token-optimizer-mcp` deserves a stronger warning than "v2, manifest costs 4-6k tokens":** the real, verified risks are (a) zero documented uninstall path in 815 lines of README, (b) a no-opt-in npm-postinstall auto-config pattern that directly edits `settings.json`, (c) a self-documented settings.json BOM-corruption bug, and (d) releases frozen for ~8 months on both npm and GitHub despite ongoing commits. The manifest-cost concern is plausible but unverified; these other four are directly sourced from the project's own README.
5. **The JetBrains 8.5%-vs-65% claim for caveman is accurate and independently reproducible** — cited above with exact methodology (SkillsBench, 82-86 paired tasks, p=0.82 on quality). Ship the warning label exactly as planned.
6. **`headroomlabs-ai/headroom`'s v2 deferral is validated** — Python runtime weight and a real paid "Desktop" competitor (`gglucass/headroom-desktop`) both confirmed.
